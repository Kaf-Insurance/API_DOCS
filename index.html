Fixed an issue where the KAF policy number wasn't being generated when submitting for issuance via the API. The policy number is now properly generated in format SME{YEAR}1{6-digit counter}.

API Response Field Additions
Added missing fields to API responses: kaf_policy_number, tpa_policy_number, payment_frequency, and all 6 reimbursement bank fields.

Fixed payment_frequency being set on the wrong model during submit-issuance. It's now correctly updated on the quotation with the same business rules as the dashboard.

What's New — March 15, 2026

Login Response Trimming (1.1)
The login response now returns a trimmed user object. Removed fields: height, weight, bmi, birth_date, gender, phone_number, fcm_token, signed_up, lang. The response now includes only essential brokerage fields: id, first_name, last_name, email, is_broker, broker_fra_id, broker_status, broker_data_confirmed, assigned_sales, assigned_sales_email, is_sales.

Purchase Request Response Cleanup
All OCR/AI-related fields have been removed from purchase request responses: *_is_ai_autofilled, *_ocr_data, *_ocr_*_token_count, *_ocr_cost, *_ocr_timestamp, *_ocr_auth_* fields for both tax_card and commercial_registry. Responses now include only the cleaned data values.

Excel Download Fixes (8.3, 8.9)
Fixed corrupted binary responses for member downloads and invoice breakdown Excel files. Both endpoints now use proper BytesIO encoding with Content-Transfer-Encoding: binary headers for reliable file delivery.

Endorsement Request Refactoring (9.4–9.8)
Major ER endpoint restructuring:

  • 9.4 Cancel ER — New endpoint POST /endorsement-requests/{er_id}/cancel/ replaces the old Update ER endpoint. Cancels a draft ER (changes status from draft to cancelled). Returns full ER object.
  • 9.5 Add Item — Now accepts category names (e.g., "K100", "100") in addition to integer IDs for medical_category_id and life_category_id. Names are automatically resolved to IDs.
  • 9.6 Export Template — New endpoint GET /endorsement-requests/{er_id}/export-template/ downloads an Excel template pre-populated with all active policy members. Pending name/phone updates in the ER are reflected in the export. Category columns use names, not IDs.
  • 9.6 Import Excel — New endpoint POST /endorsement-requests/{er_id}/import/ accepts a filled Excel template and bulk-creates ER items. Rows with matching National IDs and changed name/phone become update items; new rows become addition items; unchanged rows are skipped. Returns counts and the updated ER object.
  • 9.7 Remove Item — URL changed to /endorsement-requests/{er_id}/items/{item_id}/ (includes ER ID). Now returns the updated ER object (was 204 No Content). When all items are removed, er_type resets to addition.

ER Type Determination Fix
The determine_er_type() method now correctly returns addition (not mixed) when an ER has zero items, ensuring proper type reset after removing all items.

Policy Management - Members Section (8.1–8.4)
New endpoints for managing policy members after TPA activation. Includes viewing active members, viewing all members (active and inactive), and exporting active members to Excel. All member endpoints require the policy status to be tpa_services_activated.

Policy Management — Endorsement Requests Section (9.1–9.8)
New set of endpoints for creating and managing Endorsement Requests (ERs). All endpoints require purchase_request.status = tpa_services_activated. Key features and validations:

  • 9.1 List ERs — returns all ERs for a policy, newest first; optional ?status= filter.
  • 9.2 Create ER — creates a draft ER. Open-ER gate: blocked if any ER in draft, pending, or partially_approved (with pending items) already exists. Error response includes existing_er_id and existing_er_status.
  • 9.3 Get ER Details — returns full ER with all nested items including temp_member_data, update_fields_data, approval_status, document_url, and per-item rejection reasons.
  • 9.4 Cancel ER — cancels a draft ER (sets status to cancelled). Only works while status is draft.
  • 9.5 Add Item — adds one item of type addition, deletion, or update. Enforces at creation time:
    • Deletion: policy_member must be is_active=True and belong to this policy.
    • Addition — categories: medical_category_id and life_category_id accept either integer IDs or category name strings (e.g. "K100"). If member_type is spouse or child, life_category_id is stripped to null.
    • Update: only name and mobile_number editable; name regex enforced; mobile 11-digit normalised; old values auto-filled; name change auto-sets required_document_type = National ID.
  • 9.6 Export Template / Import Excel — download an Excel template pre-populated with active members, edit it offline, then re-import to bulk-create addition and update items.
  • 9.7 Remove Item — removes an item from a draft ER; er_type recalculated. When all items are removed, er_type resets to addition. Returns the updated ER object.
  • 9.8 Submit ER — runs full validation chain before setting status → pending. Validation gates in order:
    1. ER must not be empty (≥ 1 item).
    2. Update items gate: old ≠ new for every field.
    3. Employee deletion cascade gate: if an employee is being deleted, all of their active dependents (matched by staff_id exact or staff_id prefix e.g. EMP001-1) must also be in the deletion list. Returns 400 with each employee name and the list of missing dependents.
    4. Addition data validation: NID format + DOB cross-check; no duplicate NIDs within ER or against active members; staff ID rules; 11-digit mobile.
    5. Medical category plan restriction: medical_category_id must be one already used by the policy's active members. If no members have categories yet (edge case), check is skipped.
    6. Life category normalisation: any life_category_id on a spouse/child item is set to null.
    7. Document requirements (when today > inception_date + 30 days): spouse → Marriage Certificate required; child born ≤ 30 days → allowed; child born > 30 days → ER blocked; name-change update → National ID required.

On successful submit: status → pending, submitted_at = now(), GlobeMed alteration email sent automatically.

TPA Activation Validation (Improved)
Member list, member detail, member download, and Endorsement Request endpoints now return clear, descriptive errors when TPA status requirements are not met. Returns 404 if the policy does not exist or does not belong to the broker; returns 403 with a message like "This action requires TPA services to be activated on the policy. Current policy status: "Issued"." if the policy exists but has the wrong status.

Enhanced Member Filtering
Active members endpoint (/policies/{id}/members/) now returns only active members by default. New all-members endpoint (/policies/{id}/members/all/) supports optional filtering by active status.

3. Quotations

Quotations are created from a Lead with an Excel file containing member data. A premium calculation is automatically generated. The quotation starts in draft status.

Excel File Columns:

What happens on creation:

  1. Excel is parsed and validated (names, DOBs, categories, member types, age limits).
  2. Premiums are calculated based on category age ranges effective at the inception date.
  3. A quotation is created with all members and premium totals.
  4. The quotation starts in draft status with payment_frequency defaulting to annual.
  5. Existing lead documents (Tax ID, Commercial Registry, KYC) are copied to the quotation.
  6. Lead fields (signature, contact info, address) are copied to the quotation.

Success: 201 Created — returns the full quotation object.

3.6 Update Members

PATCH /purchase-requests/{id}/members/

application/json — Only 3 fields can be updated: national_id, staff_id, mobile_number.

{
  "members": [
    {
      "id": 1234,
      "national_id": "28505031234568",
      "staff_id": "EMP001",
      "mobile_number": "01012345678"
    }
  ]
}

Status-based rules:

  • Draft: All 3 fields can be freely set, changed, or cleared.
  • Pending Client Approval: Only empty fields can be filled. Already-filled fields are locked.
  • Other statuses: Updates are completely blocked.

Validations Applied:

  • National ID: 14 digits for Egypt, min 4 characters for other countries, DOB cross-check for Egypt
  • Duplicate Detection: No duplicate National IDs within the same policy
  • Staff ID: Required for all members, unique for employees, auto-suffixed for dependents
  • Mobile Number: 11 digits, auto-prefixed with 0 if 10 digits provided

Success: 200 OK — {"message": "Successfully updated 2 members", "updated_count": 2}

Error Response (400 Bad Request):

{
  "error": "National ID validation failed for some members",
  "details": [
    "Update 1: Ahmed Mohamed - National ID must be exactly 14 digits for Egypt"
  ]
}

9.5 Add Item to Endorsement Request

POST /endorsement-requests/{er_id}/items/

Adds a single item to a draft ER. Supports addition, deletion, and update action types. Accepts JSON or multipart/form-data (required when attaching a document).

After every item is added, the ER's er_type is auto-recalculated (addition / deletion / update / mixed).

Addition item:

{
  "action_type": "addition",
  "temp_member_data": {
    "name": "Sara Ahmed",
    "date_of_birth": "2000-01-01",
    "member_type": "employee",
    "national_id": "30001011234567",
    "staff_id": "EMP005",
    "mobile_number": "01012345678",
    "country": "Egypt",
    "medical_category_id": "K100",
    "life_category_id": "100"
  },
  "effective_date": "2026-04-01"
}

Data stored as-is. Deep validation (NID, duplicates, staff IDs, document rules, medical category plan check) runs on submit, not at item creation.

Category fields: medical_category_id and life_category_id accept either integer IDs or category name strings (e.g. "K100", "100"). Names are resolved to IDs automatically.

Life category normalisation (enforced at creation): if member_type is spouse or child, any supplied life_category_id is stripped to null before storage.

Deletion item:

{
  "action_type": "deletion",
  "policy_member": 45,
  "effective_date": "2026-04-01"
}

Validations (deletion): policy_member must be is_active=True and belong to this policy.

Employee cascade (enforced at submit, not at creation): if this member is an employee, all of their active dependents must also be added as deletion items before the ER can be submitted.

Update item (non-transactional):

{
  "action_type": "update",
  "policy_member": 22,
  "update_fields_data": {
    "name": { "old": "Ahmed Ali", "new": "Ahmed Ali Hassan" },
    "mobile_number": { "old": "01012345678", "new": "01098765432" }
  }
}

Editable fields: name and mobile_number only. All other fields require deletion + re-addition.

Validations (update):

  • policy_member must be active and belong to this policy
  • Only name and mobile_number keys allowed — any other key returns 400
  • name: non-empty; Arabic/English letters, spaces, hyphens, apostrophes only
  • mobile_number: digits only; 10-digit input auto-prefixed with 0; must be 11 digits
  • Old values auto-filled from current member if not provided
  • Name change → required_document_type auto-set to National ID — must upload National ID document before ER can be submitted

Error Responses:

  • 400 — ER not in draft, invalid action_type, invalid category name, validation failure
  • 404 — ER not found, policy_member not found or not active

9.8 Submit Endorsement Request

POST /endorsement-requests/{er_id}/submit/

Submits a draft ER for operations team review. Runs full validation before changing status to pending.

Validation Gates (in order):

  1. Not empty — ER must have ≥ 1 item
  2. Update items gate — every update item's update_fields_data must be non-empty and old ≠ new for each field
  3. Employee deletion cascade gate — for every deletion item targeting an employee, all of that employee's active dependents (spouse/child whose staff_id equals the employee's staff_id exactly, or starts with "{staff_id}-", e.g. EMP001-1, EMP001-2) must also be present as deletion items in this ER. Returns 400 with a details list per employee showing employee_name, employee_id, and missing_dependents (id, name, member_type).
  4. Addition data validation:
    • Required fields: name, date_of_birth, national_id, member_type
    • Egyptian NID: 14 digits; DOB encoded in NID must match date_of_birth (century code: 2→1900s, 3→2000s)
    • Non-Egyptian: passport/ID ≥ 4 characters
    • No duplicate NIDs within ER additions
    • No NID already in active policy members
    • staff_id required; employee IDs unique; dependent IDs reference an existing employee and are auto-suffixed (EMP001-1, EMP001-2, …)
    • mobile_number: 10 or 11 digits (10-digit auto-prefixed with 0)
    • Medical category plan restriction: medical_category_id must be one of the category IDs already in use by the policy's existing active members. If the policy has no members with categories yet (edge case), this check is skipped — mirrors the frontend dropdown which is pre-filtered to the policy's plans.
    • Life category normalisation: any life_category_id present on a spouse or child addition item is set to null — dependents cannot hold a life insurance category.
  5. Document requirements (only when today > inception_date + 30 days):
    • Spouse addition → Marriage Certificate upload required
    • Child born ≤ 30 days ago → no document needed
    • Child born > 30 days ago → ER BLOCKED entirely — broker must contact operations team
    • Update item with name change → National ID document required

On success: status → pending, submitted_at = now(), GlobeMed alteration email sent.

Success Response (200 OK): Returns the full updated ER object (same structure as 9.3).

Error Responses:

  • 400 — Validation failed (NID errors, duplicates, missing staff ID, invalid medical category, missing dependents in employee deletion, missing documents, blocked child addition, etc.) — includes details array
  • 404 — ER not found, not in draft, or does not belong to you

Endorsement Request Workflow

How to create and submit an ER to add, remove, or update members on a tpa_services_activated policy.

Lifecycle Steps

  1. Create ER (9.2) — POST /policies/{id}/endorsement-requests/ Blocked by open-ER gate if a draft/pending/partially_approved ER already exists · Returns existing_er_id and existing_er_status on block
  2. Add items individually (9.5) — or bulk via Excel import (step 3) POST /endorsement-requests/{er_id}/items/ · Supports action_type: addition, deletion, update
  3. Export template + edit + import (9.6) — alternative bulk approach GET export-template/ → edit Excel offline → POST import/ · Additions for new rows · Updates for changed name/phone · Unchanged rows skipped
  4. Review ER items (9.3) — verify all items are correct GET /endorsement-requests/{er_id}/ · Remove individual items via 9.7 if needed · er_type recalculates automatically
  5. Submit ER (9.8) — runs all 7 validation gates POST /endorsement-requests/{er_id}/submit/ · Status → pending · GlobeMed alteration email sent automatically
  6. (If needed) Cancel ER (9.4) — only available while status is draft POST /endorsement-requests/{er_id}/cancel/ · Status draft → cancelled · Once submitted (pending) cancellation is not possible

ER Item Types

Addition

Add a new member. Provide full temp_member_data: name, DOB, NID, member_type, medical_category_id (name or ID accepted), country, staff_id, mobile. Validated deeply at submit.

Deletion

Remove an active member. Provide policy_member ID (must be active and belong to this policy). Employee cascade rule: all active dependents (matched by staff_id) must also be in the deletion list.

Update

Update name or mobile_number of an active member. Old values are auto-filled from the current record. Name change auto-sets required_document_type = National ID.

Submit Validation Gates (9.8)

All 7 gates must pass in order for the ER to be submitted:

  • Gate 1
    Not Empty — ER must have at least 1 item. Returns 400: Cannot submit an empty endorsement request
  • Gate 2
    Update Items Valid — every update item must have update_fields_data with old ≠ new for each field included
  • Gate 3
    Employee Deletion Cascade — if an employee is being deleted, all their active dependents (matched by staff_id prefix, e.g. EMP001-1) must also have deletion items. Returns list of missing dependents per employee.
  • Gate 4
    Addition Data Validation — NID 14-digit format + DOB cross-check; no duplicate NIDs within ER or against active policy members; staff ID uniqueness; 11-digit mobile number
  • Gate 5
    Medical Category Restriction — medical_category_id for additions must already be used by at least one active policy member (skipped if policy has no categorised active members yet)
  • Gate 6
    Life Category Normalisation — life_category_id on any spouse or child addition item is automatically set to null
  • Gate 7
    Document Requirements (applies when today > inception_date + 30 days): spouse → Marriage Certificate required; child born ≤ 30 days → allowed; child born > 30 days → ER blocked entirely; name-change update → National ID document required

Excel Export / Import Workflow

Export Template GET /endorsement-requests/{er_id}/export-template/ → downloads .xlsx pre-filled with all active members
Edit Offline add new rows for additions · change Name or Phone Number on existing rows for updates · leave others unchanged
Import Excel POST /endorsement-requests/{er_id}/import/ · multipart field: file
New NID rows → addition items created

Import appends new items — it does NOT clear existing ER items. It can be called multiple times. Response includes additions_created, updates_created, rows_skipped counts, and the updated ER object.

Invoice Settling Workflow

Invoices are generated by admin and visible to the broker from pending_issuance status onwards.

Settling Steps

  1. List Invoices (8.5) — review all invoices and their settlement status GET /policies/{id}/invoices/ · Includes settlement_status: unsettled / pending / settled
  2. Get Invoice Details (8.6) — check amount, due date, breakdown GET /policies/{id}/invoices/{invoice_id}/
  3. Download Invoice PDF (8.8) — optional, review the document GET /policies/{id}/invoices/{invoice_id}/download/
  4. Download Members Breakdown (8.9) — optional, per-member premium Excel GET /policies/{id}/invoices/{invoice_id}/breakdown/ · Returns .xlsx with Medical + Life premium per member
  5. Settle Invoice (8.10) — upload proof of payment POST /policies/{id}/invoices/{invoice_id}/settle/ · multipart: proof_of_payment (PDF/JPG/PNG) + optional amount + collection_date · settlement_status → pending
  6. Admin confirms — settlement_status changes to settled No broker action needed · Poll 8.7 Check Invoice Status to track confirmation

Document Auto-Propagation

When a document is uploaded to a Lead (either during creation or via individual upload endpoints), it is automatically copied to all quotations of that lead that are in draft or Pending Client Approval status. Quotations in pending_issuance or later statuses are not affected.
